## 開發規則

### 閱讀 README 優先

每次改 code 之前，**必須**先閱讀 `README.md` 了解檔案架構與專案結構，而不是重新掃描整個專案浪費 token。



### 格式化

每次修改完 Dart 檔案後，**必須**在 commit 前執行：

```bash
dart format .
```

CI 會用 `dart format --output=none --set-exit-if-changed .` 檢查格式，未格式化的檔案會導致 CI 失敗。

## Karpathy 行為準則

> 來源：[forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)
> 降低常見 LLM 編碼錯誤的行為準則。**取捨：** 這些準則偏向謹慎而非速度，對於瑣碎任務請自行判斷。

### 1. Think Before Coding

**不要假設。不要隱藏困惑。說出取捨。**

實作前：
- 明確陳述你的假設。不確定時，請問。
- 若存在多種解讀，列出來——不要自行默默選擇。
- 若有更簡單的做法，說出來。必要時推回去。
- 若有不清楚的地方，停下來。說明是什麼讓你困惑。問清楚。

### 2. Simplicity First

**解決問題所需的最少程式碼。不要有推測性的內容。**

- 不加超出要求的功能。
- 不對單次使用的程式碼做抽象化。
- 不加沒有被要求的「彈性」或「可配置性」。
- 不對不可能發生的情境做錯誤處理。
- 如果你寫了 200 行但 50 行就夠了，重寫。

自問：「資深工程師會說這太複雜嗎？」如果是，簡化。

### 3. Surgical Changes

**只碰必須碰的。只清理自己造成的亂。**

編輯現有程式碼時：
- 不要「改善」相鄰的程式碼、注解或格式。
- 不要重構沒壞的東西。
- 符合現有風格，即使你會做得不同。
- 若發現無關的死碼，提出來——不要刪除它。

你的修改造成孤兒時：
- 移除**你的修改**造成的未使用 import/變數/函式。
- 不要移除原本就存在的死碼，除非被要求。

測試標準：每一行修改都應該直接對應到使用者的要求。

### 4. Goal-Driven Execution

**定義成功標準。循環直到驗證完成。**

將任務轉換為可驗證的目標：
- 「加入驗證」→「為無效輸入寫測試，再讓測試通過」
- 「修復 bug」→「寫一個能重現 bug 的測試，再讓測試通過」
- 「重構 X」→「確保重構前後測試都通過」

對於多步驟任務，說明簡短計畫：
```
1. [步驟] → 驗證：[檢查項目]
2. [步驟] → 驗證：[檢查項目]
3. [步驟] → 驗證：[檢查項目]
```

強的成功標準讓你能獨立循環執行。弱的標準（「讓它能動」）需要不斷澄清。
